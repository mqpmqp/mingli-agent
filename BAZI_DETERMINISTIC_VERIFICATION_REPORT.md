## Recorded source conflict

At 23:00, `sxtwl` configured with `isZaoWanZiShi=false` and the declared 00:00 boundary keeps the current day's stem for the Zi-hour calculation. `lunar_python 1.4.8` keeps its own default hour-pillar behavior even when its day-pillar sect is set to 2. The fixture records this as `status=unresolved`; it does not automatically select one external result. The project's declared result is separately verified against configurable sxtwl behavior and the Hong Kong Observatory day-stem/hour-stem table.

## Known limits

- The low-precision solar-longitude implementation differs from the independent sxtwl ephemeris by at most 12.93 minutes in the supported-range audit. A 15-minute exclusion window fails explicitly rather than risking a silent boundary choice.
- The NOAA fractional-year equation of time is an approximation. Latitude is retained but not used by this method.
- Historical timezone accuracy depends on the IANA database available through the host or Windows `tzdata` package.
- Luck start age follows exactly one declared days-divided-by-three convention; alternate schools are not mixed into this method.
- The engine calculates calendar fields only. It does not calculate event outcomes and successful execution does not establish prediction validity.
